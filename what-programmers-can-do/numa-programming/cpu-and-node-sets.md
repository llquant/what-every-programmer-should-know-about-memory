# 6.5.6. CPU 与节点集合

通过将程序改变为使用目前为止所描述的接口来为 SMP 与 NUMA 环境调整程序，在来源[^译注]无法取得的情况下可能会极为昂贵（或者不可能）。此外，系统管理员可能想要对用户和／或进程可以使用的资源施加限制。对于这些情境，Linux 系统核心支持所谓的 CPU 集。这个名称有一点误导，因为 memory 节点也被涵盖其中。它们也与 `cpu_set_t` 数据类型无关。

此刻，CPU 集的接口为一个特殊的文件系统。它通常没有被挂载（mount）（至少到目前为止）。这可以使用

`mount -t cpuset none /dev/cpuset`

改变。挂载点 `/dev/cpuset` 在这个时间点当然必须存在。这个目录的内容为默认（根） CPU 集的描述。它起初由所有的 CPU 与所有的 memory 节点所构成。这个目录中的 `cpus` 文件显示在 CPU 集中的 CPU、`mems` 文件显示 memory 节点、`tasks` 文件显示进程。

为了建立一个新的 CPU 集，只要在阶层结构中的某个地方建立一个新的目录。新的 CPU 集会继承来自父集合的所有设置。接着，新的 CPU 集的 CPU 与 memory 节点可以通过将新值写到在新目录中的 `cpus` 与 `mems` 虚拟文件来更改。

若是一个进程属于一个 CPU 集，CPU 与 memory 节点的设置会被用作亲和性与 memory 策略 bit 掩码的掩码。这表示，程序无法在亲和性掩码里选择不在进程正在使用的 CPU 集（即，它在 `tasks` 文件中列出的位置）的 `cpus` 文件中的任何 CPU。对于 memory 策略的节点掩码与 `mems` 文件也是类似的。

除非 bit 掩码在掩码后为空，否则程序不会经历任何错误，因此 CPU 集是一种控制程序执行的近乎无形的手段。这种方法在有着大量 CPU 与／或 memory 节点时是尤其有效率的。将一个进程移到一个新的 CPU 集，就跟将进程 ID 写到合适 CPU 集的 `tasks` 文件一样简单。

CPU 集的目录包含许多其他文件，能用来指定像是 memory 压力下、以及独占访问 CPU 与 memory 节点时的行为。感兴趣的读者请参阅系统核心源代码树中的文件 `Documentation/cpusets.txt`。


[^译注]: 根据前后文猜测，这里的「来源」指的应该是程序使用的 CPU 与 memory 节点。
