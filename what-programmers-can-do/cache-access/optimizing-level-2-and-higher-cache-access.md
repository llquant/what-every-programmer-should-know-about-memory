# 6.2.3. 优化二级与更高层 cache 访问

关于一级 cache 的优化所说的一切也适用于二级与更高层 cache 访问。有两个最后一级 cache 的额外面向：

* cache 未命中一直都非常昂贵。L1 未命中（希望）频繁地命中 L2 与更高层 cache，于是限制其损失，但最后一级 cache 显然没有后盾。
* L2 cache 与更高层 cache 经常由多颗处理器核与／或 HT 所共享。每个执行单元可用的有效 cache 大小因而经常小于总 cache 大小。

为了避免 cache 未命中的高成本，工作集大小应该配合 cache 大小。如果数据只需要一次，这显然不是必要的，因为 cache 无论如何都没有效果。我们要讨论的是被需要不只一次的数据集的工作负载。在这种情况下，使用一个太大而不能塞得进 cache 的工作集将会产生大量的 cache 未命中，即使预取成功地执行，也会拖慢程序。

即使数据集太大，一个程序也必须完成它的职责。以最小化 cache 未命中的方式完成工作是程序员的职责。对于最后一级 cache，是可能——如同 L1 cache——以较小的部分来执行工作的。这与表 6.2 优化的矩阵乘法非常雷同。不过，有一点不同在于，对于最后一级 cache，要处理的数据块可能比较大。如果也需要 L1 优化，程序会变得更加复杂。想象一个矩阵乘法，其数据集——两个输入矩阵与输出矩阵——无法同时塞进最后一级 cache。在这种情况下，或许适合同时优化 L1 与最后一级 cache 访问。

众多处理器世代中的 L1 cache 行大小经常是固定的；即使不同，差异也很小。假设为较大的大小是没什么大问题的。在具有较小 cache 大小的处理器中，会用到两个或更多 cache 行、而非一个。在任何情况下，写死 cache 行大小、并为此优化程序都是合理的。

对于较高层次的 cache，若程序是假定为一般化的话，就不是这样。那些 cache 的大小可能有很大的差异。八倍或更多倍并不罕见。将较大的 cache 大小假定为默认大小是不可能的，因为这可能表示，除了那些具有最大 cache 的机器之外，程序在所有机器上都会表现得很差。相反的选择也很糟：假定为最小的 cache，代表浪费掉 87% 或者更多的 cache。这很糟；如同我们能从图 3.14 看到的，使用大 cache 对程序的速度具有巨大的影响。

这表示程序必须动态地将自身调整为 cache 行大小。这是一种程序特有的优化。我们这里能说的是，程序员应该正确地计算程序的需求。不仅数据集本身需要，更高层次的 cache 也会被用于其他目的；举例来说，所有执行的指令都是从 cache 加载的。如果使用函数库中的函数，这种 cache 的使用可能会加总为一个可观的量。那些函数库函数也可能需要它们自己的数据，进一步减少可用的内存。

一旦我们有一个内存需求的公式，我们就可以将它与 cache 大小作比较。如同先前所述，cache 可能会被许多其他处理器核所共享。目前[^34]，在没有写死知识的情况下，获取正确信息的唯一方法是通过 `/sys` 文件系统。在表 5.2，我们已经看过内核发布的有关于硬件的信息。程序必须在目录：

`/sys/devices/system/cpu/cpu*/cache`

找到最后一级 cache。这可以由在这个目录里的层次文件中的最高数值来辨别出来。当目录被识别出来时，程序应该读取在这个目录中的 `size` 文件的内容，并将数值除以 `shared_cpu_map` 文件中的 bit 掩码中设置的数字。

以这种方式计算的值是个安全的下限。有时一个程序会知道多一些有关其他线程或进程的行为。如果那些线程被调度在共享这个 cache 的处理器核或 HT 上、并且已知 cache 的使用不会耗尽它在总 cache 大小中所占的那份，那么计算出的限制可能会太小，而不是最佳的。是否要比公平共享应该使用的还多，真的要视情况而定。程序员必须做出选择，或者必须让用户做个决定。



[^34]: 当然很快就会有更好的方法！
