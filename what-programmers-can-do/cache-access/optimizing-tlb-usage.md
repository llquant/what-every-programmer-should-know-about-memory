# 6.2.4. 优化 TLB 使用

有两种 TLB 使用的优化。第一种优化是减少一个程序必须使用的页数。这会自动导致较少的 TLB 未命中。第二种优化是通过减少必须被分配的较高层目录表的数量，以令 TLB 查询便宜一些。较少的目录表代表使用的内存较少，这可能使得目录查询有较高的 cache 命中率。

第一种优化与页错误的最小化密切相关。我们将会在 7.5 节仔细地涵盖这个主题。页错误经常是个一次性的成本，但由于 TLB cache 通常很小而且会被频繁地刷新，因此 TLB 未命中是个长期的损失。页错误比起 TLB 未命中还贵了数个数量级，但如果一个程序跑得足够久、而且程序的某些部分会被足够频繁地执行，TLB 未命中甚至可能超过页错误的成本。因此重要的是，不仅要从页错误的角度、也要从 TLB 未命中的角度来考虑页优化。差异在于，页错误的优化只要求页范围内的代码与数据分组，而 TLB 优化则要求——在任何时间点——尽可能少的 TLB 项目。

第二种 TLB 优化甚至更难控制。必须使用的页目录数量是视进程的虚拟地址空间中使用的地址范围分布而定的。地址空间中广泛多样的位置代表着更多的目录。

一个难题是，地址空间布局随机化（Address Space Layout Randomization，ASLR）恰好造成这种状况。栈、DSO、堆、与可能的可执行文件的加载地址会在运行时随机化，以防止机器的攻击者猜出函数或变量的地址。

只有在最大性能至关重要的情况下，才应该关闭 ASLR。额外目录的成本低到足以令这步是不必要的，除了一些极端的状况之外。内核能随时执行的一个可能的优化是，确保一个单一的映射不会横跨两个目录之间的内存空间边界。这会以最小的方式限制 ASLR，但不足以大幅地削弱它。

程序员直接受此影响的唯一方式是在明确请求一个地址空间区域的时候。这会在以 `MAP_FIXED` 使用 `mmap` 的时候发生。以这种方式分配地址空间区域非常危险，人们几乎不会这么做。如果程序员使用上述方法且允许自由选取地址，则他们应该要知道最后一级页目录的边界，及适当挑选所请求的地址。
