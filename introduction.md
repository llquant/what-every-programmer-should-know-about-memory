# 1. 引言

早期的计算机简单许多。系统中的各个组件，例如 CPU、内存、大容量存储设备（mass storage）以及网络接口（network interface），是共同发展起来的，因此性能相当均衡。举例来说，在提供数据方面，内存和网络接口并不比 CPU 快多少。

随着计算机基本结构稳定下来，硬件开发者开始专注于优化各个独立子系统，情况也随之改变。某些计算机组件的性能突然明显落后，并形成瓶颈。大容量存储设备和内存子系统尤其如此：出于成本原因，它们相对于其他组件进步得更慢。

大容量存储设备速度慢的问题主要通过软件技术处理：操作系统（operating system）会把最常使用（也最可能被使用）的数据保存在主内存（main memory）中，而主内存的访问速度比硬盘（hard disk）快几个数量级。存储设备本身也加入了缓存，这种做法不需要修改操作系统就能提升性能。[^1]由于偏离本文主旨，我们不会继续深入讨论大容量存储访问的软件优化细节。

与存储子系统不同，消除主内存瓶颈已经被证明要困难得多，而且几乎所有解决方案都需要修改硬件。如今这些修改主要有以下几种形式：

* RAM 的硬件设计（速度以及并行度〔parallelism〕）。
* 内存控制器（controller）的设计。
* CPU 缓存。
* 设备的直接内存访问（Direct Memory Access，DMA）。

本文主要讨论 CPU 缓存，以及内存控制器设计带来的一些影响。在探索这些主题的过程中，我们也会把 DMA 放入更大的图景中理解。不过，我们会先概览当今商用硬件的设计；这是理解如何高效使用内存子系统及其限制的必要前提。我们也会稍加详细地介绍不同类型的 RAM，并说明为什么这些差异至今仍然存在。

这篇文档绝不是包罗万象的最终版本。本文仅限于商用硬件，而且还进一步限于这类硬件的一个子集。同时，许多主题只会讨论到足以满足本文目标的程度。对于这些主题，建议读者查阅更详细的文档。

涉及特定操作系统的细节与解决方案时，本文只讨论 Linux。文中不会包含任何其他操作系统的信息。作者无意讨论其他操作系统的影响。如果读者认为自己必须使用其他操作系统，就应该要求对应厂商撰写类似本文的文档。

开始前还有最后一点说明。本文多次使用“通常”以及其他类似的限定语。这里讨论的技术在现实世界中有大量变种，而本文只涉及最常见、最主流的版本。很少能对这些技术作出绝对判断，所以需要这些限定语。

## 文档结构

本文主要写给软件开发者，不会深入到足以满足硬件方向读者需求的技术细节。但在讨论对开发者实用的信息之前，还必须先打下不少基础。

为此，第二节会从技术细节上描述随机访问内存（Random-Access Memory，RAM）。这一节值得了解，但并不是理解后续章节的绝对必要条件。凡是需要用到本节内容的地方，都会加入适当的反向引用（back reference），所以心急的读者一开始可以跳过本节的大部分内容。

第三节详细描述 CPU 缓存的行为。文中会使用一些图表，以免内容变得过于枯燥。本节内容是理解本文后续章节所不可或缺的。第四节简要描述虚拟内存（virtual memory）的实现方式，这也是后续内容所需的基础。

第五节详细描述非均匀内存访问（Non-Uniform Memory Access，NUMA）系统。

第六节是本文的核心章节。它汇总前面几节的信息，并向程序员说明如何在各种场景中编写性能良好的代码。非常心急的读者可以从这一节开始，并在必要时回到前面的章节复习底层技术知识。

第七节介绍一些能帮助程序员把工作做得更好的工具。即使完全理解了这些技术，要在非平凡的软件项目中明确问题所在也远非显而易见；因此，一些工具是必要的。

最后在第八节，我们将展望未来几年可以期待的技术，以及一些也许只是值得拥有的技术。
