# 3.4.1. 自我修改的代码

在计算机时代的早期，内存是很珍贵的。人们不遗馀力地减少程序的大小，以为程序数据腾出更多的空间。一个经常使用的技巧是，随着时间改变程序自身。偶尔仍旧会找到这种自我修改的代码（Self Modifying Code，SMC），如今多半是为了性能因素、或者用在安全漏洞上。

一般来说应该避免 SMC。虽然它通常都被正确地执行，但具有并非如此的边界案例（boundary case），而且没有正确完成的话，它会产生性能问题。显然地，被改变的代码无法维持在保存被解码指令的追踪 cache 中。但即使代码完全（或者有时候）不会被执行，因而不会使用到追踪 cache，处理器也可能会有问题。如果接下来的指令在它已经进入流水线的期间被改变，处理器就得丢掉大量的成果，然后从头开始。甚至有处理器的大多数状态都必须被丢弃的情况。

最后，由于处理器假定——为了简化起见，而且因为这在 99.9999999% 的情况下都成立——代码页是不可修改的（immutable），所以 L1i 的实现不会采用 MESI 协议，而是一种简化的 SI 协议。这表示，如果侦测到修改，就必须做出许多的悲观假设。

强烈地建议尽可能避免 SMC。内存不再是如此稀有的资源。最好是撰写各自的函数，而非根据特定的需求修改一个函数。或许有天 SMC 的支持可以是可选的，而我们就可以用这种方式侦测出尝试修改代码的漏洞代码（exploit code）。如果真的必须使用 SMC，写入操作应该要绕过 cache，以免因为 L1i 所需的 L1d 的数据造成问题。关于这些指令的更多消息，见 6.1 节。

在 Linux 上，识别出包含 SMC 的程序通常非常容易。使用正规工具链（toolchain）构建的话，所有程序的代码都是防写的（write-protected）。程序员必须在链接时（link time）施展强大的魔法，以产生程序页可以被写入的可执行文件。当这种情况发生时，现代的 Intel x86 与 x86-64 处理器具有专门的、计算自我修改的代码使用次数的性能计数器。有了这些计数器的帮助，非常轻易就可以识别具有 SMC 的程序，即使程序由于宽松的许可而成功执行。
