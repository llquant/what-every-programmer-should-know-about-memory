# 3.3.5. 其他细节

到目前为止，我们已经讨论过由三个部分 –– 标签、集合索引、以及 cache 行偏移量 –– 组成的地址。但实际使用的地址是什么呢？所有有关的处理器如今都是将虚拟地址空间提供给进程，这代表有两种不同的地址：虚拟的以及物理的。

虚拟内存的问题是，它们不是唯一的。虚拟内存可以 –– 随着时间 –– 指涉到不同的物理内存地址。在不同进程中的相同地址也可能会指涉到不同的物理地址。所以使用物理内存地址永远比较好，对吧？

这里的问题是，在运行期间使用的虚拟内存必须在内存管理单元（Memory Management Unit，MMU）的帮助下转换成物理地址。这是个不单纯的操作。在执行一个指令的管线中，物理地址可能只有在之后的阶段才能取得。这表示 cache 逻辑必须非常快速地判定这个内存位置是否被 cache 了。若是可以使用虚拟地址，cache 查询就可以早点在管线中进行，并且在 cache 命中的情况下，就可以取得内存内容了。结果是，管线可以隐藏更多内存访问的成本。

处理器设计者目前是使用虚拟地址来标记第一层次的 cache。这些 cache 非常地小，而且清除也不会太费力。若是一个进程的页表树（page table tree）改变了，至少必须局部地清理 cache。假如处理器拥有可以指定被改变的虚拟地址范围的指令，就有可能避免一次完整的刷新。考虑到 L1i 与 L1d cache 的等待时间很短（~3 周期），使用虚拟地址几乎是强迫性的。

对于大一点的 cache，包含 L2、L3、... cache，是需要物理地址标记的。这些 cache 有比较长的等待时间，而虚拟→物理地址转换可以在时间内完成。因为这些 cache 比较大（即，当它们被刷新时，会损失大量的信息），并且因为主内存访问的等待时间，重新填入它们会花上很久的时间，因此刷新它们的代价经常不小。

一般来说，应该没必要知道在那些 cache 中的地址管理的细节。它们无法改变，而且所有会影响性能的因子通常是应该避免、或者是与高成本相关联的东西。塞满 cache 容量是很糟的，而且若是多数被使用的 cache 行都落在同一组集合中，所有 cache 都会很快地碰到问题。后者可以用虚拟寻址的 cache 来避免，但对于用户层次的进程来说，要避免使用物理地址寻址的 cache 是不可能的。或许唯一应该记住的细节是，可能的话，别在同个进程里将同个物理内存位置映射到两个以上的虚拟地址。

另一个对程序开发者来说满无趣的 cache 细节是 cache 的替换策略。大多 cache 会先逐出近期最少使用的（Least Recently Used，LRU）元素。以较大的关联度（由于增加更多的处理器核，关联度可能会在接下来几年内进一步地增长）维持 LRU 清单会变得越来越昂贵，我们可能会看到被采用的不同策略。

至于 cache 的替换，一介程序开发者能做的不多。若是 cache 是使用物理地址的标签，就没有任何办法能找出虚拟地址与 cache 集之间的关联。所有逻辑页的 cache 行可能都映射到相同的 cache 集，留着大量的 cache 不用。如果是这样的话，不让这太常发生就是操作系统的工作了。

随着虚拟化（virtualization）的出现，事情变得更加复杂。现在甚至不是操作系统拥有对物理内存指派的控制。而是虚拟机器监视器（Virtual Machine Monitor，VMM，又称 Hypervisor）有指派物理内存的职责。

一介程序开发者能做的最多就是 a) 完全使用逻辑内存页 b) 使用尽可能大的页大小，以尽可能地多样化物理地址。较大的页大小也有其他好处，但这是另一个主题（见第四节）。
