# 3.3.3. 写入行为

在我们开始研究在多执行环境（execution context）（线程或进程）使用相同内存的 cache 行为之前，我们必须先探究一个 cache 实现的细节。cache 是假定为一致的（coherent），而且对用户层次的程序而言，这个一致性是假定为完全透明的。系统核心程序是不同的情况；它偶尔会要求 cache 刷新（flush）。

这具体意味着，假如一个 cache 行被修改了，在这个时间点之后，对系统而言的结果与根本没有 cache、并且是主内存位置本身被修改的情况是相同的。这能以两种方式或策略来实行：

* 直写式（write-through）cache 实现；
* 回写式（write-back）cache 实现。

直写式 cache 是最简单的 cache 一致性的实行方式。若是 cache 行被写入的话，处理器也会立即将 cache 行写到主内存中。这保证了主内存与 cache 永远保持一致。可以在任何 cache 行被取代的时候直接丢弃 cache 的内容。这个 cache 策略很简单，但并不是非常快。举例来说，一支不断地修改一个区域变量的程序会在 FSB 产生大量的流量，尽管数据很可能不会在别处用到、而且可能只会短暂存在。

回写式策略更为复杂。这时处理器不会立即将被修改的 cache 行写回到主内存里。取而代之地，cache 行只会被标记为脏的。当 cache 行在未来的某个时间点从 cache 被丢弃时，脏位（dirty bit）将会在这时通知处理器去把数据写回去，而不是直接丢弃内容。

写回式 cache 有机会做得好非常多，这就是大多有着像样处理器的系统中，内存都会以这种方式 cache 的原因了。处理器甚至能在 cache 行必须被清除之前，利用 FSB 的闲置容量来存储 cache 行的内容。这使得脏位被清除，并且在需要 cache 中的空间的时候，处理器可以直接丢弃这个 cache 行。

但回写式实现也有个重大的问题。当有多于一个处理器（或是处理器核或 HT），并且访问到同样的内存时，它仍旧必须保证每个处理器看到的一直都是相同的内存内容。假如一个 cache 行在一个处理器上是脏的（也就是说，它还没被写回去），并且第二个处理器试着读取相同的内存位置，这个读取操作就不能直接送到主内存去。而是需要第一个处理器的 cache 行的内容。在下一节，我们将会看到这在当前是如何实现的。

在此之前，还有两种 cache 策略要提一下：

* 合并写入（write-combining）；以及
* 不可 cache（uncacheable）

这两种策略都是用在地址空间中、并非被真正的 RAM 所支持的特殊区域。系统核心为这些地址范围设置了这些策略（在使用了内存类型范围寄存器〔Memory Type Range Register，MTRR〕的 x86 处理器上），剩下的部分自动地进行。MTRR 也能用于在直写式与回写式策略之间选择。

合并写入是一种受限的 cache 优化，更常用于显示卡一类装置上的 RAM。由于对装置来说，传输成本比本地 RAM 访问的成本还高得多，因此避免过多的传输是更为重要的。仅因为 cache 行中的一个 word 被修改，就传输一整个 cache 行，在下一个操作修改了下一个 word 的情况下是很浪费的。可以轻易地想象，一种常见的情况是，表示屏幕上水平相邻的像素点的内存，在多数情况下也是相邻的。如同名字所暗示的，合并写入会在 cache 行被写出去之前合并多个写入访问。在理想情况下，cache 行会被一个 word 接着一个 word 地修改，并且只有在写入最后一个 word 之后，cache 行才会被写到装置中。这可以显着地加速装置对 RAM 的访问。

最后，不可 cache 的内存。这通常表示内存位置根本不被 RAM 所支持。它可能是一个被写死的特殊地址，以拥有某个在 CPU 外部实现的功能。对商用硬件来说，最常见的例子是内存映射的（memory-mapped）地址范围，转换为对连接到总线上的扩充卡以及装置（PCIe 等等）的访问。在嵌入式单板（embedded board）上，有时候会发现可以用来开关 LED 的内存地址。cache 这类地址显然是个坏点子。在这种情境下的 LED 是用以除错或者状态回报的，会想要尽可能快地看到它。在 PCIe 扩充卡上的内存可以在不与 CPU 互动的情况下改变，因此这种内存不应该被 cache。
