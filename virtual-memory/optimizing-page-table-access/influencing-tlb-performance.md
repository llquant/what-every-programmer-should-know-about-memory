# 4.3.2. 影响 TLB 性能

有几个影响 TLB 性能的因素。第一个是页的大小。显然地，页越大、会被塞进去的指令或数据对象也越多。所以一个比较大的 cache 大小减少了所需地址转换的整体数量，代表 TLB cache 中需要更少的项目。大部分架构如今允许使用多种不同的页大小；有些大小可以并存地使用。举例来说，x86／x86-64 处理器拥有寻常的 4kB 页大小，但它们也分别可以使用 4MB 与 2MB 的页。IA-64 与 PowerPC 允许像是 64kB 的大小作为基础页大小。

不过，大页尺寸的使用也随之带来了一些问题。为了大页而使用的 memory 区域在物理 memory 中必须是连续的。若是物理 memory 管理的单位大小被提高到虚拟 memory 页的大小，浪费的 memory 总量就会增加。各种 memory 操作（像是加载可执行程序）需要对齐到页边界。这表示，平均而言，在每次映射的物理 memory 中，每次映射浪费了一半的页大小。这种浪费可以轻易地累加；这因此对物理 memory 分配的合理单位大小加了个上限。

将单位大小提升到 2MB，以容纳 x86-64 上的大页无疑并不实际。这个大小太大了。但这又意味着每个大页必须由多个较小的页所构成。而且这些小页在*物理*memory 中必须是连续的。以 4kB 的单位页大小分配 2MB 的连续物理 memory 具有挑战性。这需要寻找一个有着 512 个连续页的空闲区域。在系统执行一段时间、并且物理 memory 变得片段之后，这可能极端困难（或者不可能）。

因此在 Linux 上，有必要在系统启动的时候使用特殊的 `hugetlbfs` 文件系统来分配这些大页。一个固定数量的物理页会被保留来专门作为大虚拟页来使用。这绑住了可能不会一直用到的资源。这也是个有限的池（pool）；增加它通常代表着重新启动系统。尽管如此，在性能贵重、资源充足、且麻烦的设置不是个大阻碍的情况下，庞大的页便为大势所趋。数据库服务器就是个例子。

提高最小的虚拟页大小（对比于可选的大页）也有它的问题。memory 映射操作（例如，加载应用程序）必须遵循这些页大小。不可能有更小的映射。一个可执行程序不同部分的位置 –– 对大多架构而言 –– 有个固定的关系。若是页大小增加到超过在可执行程序或者 DSO 创建时所考虑的大小时，就无法执行加载操作。将这个限制记在心上是很重要的。图 4.3 显示了可以如何决定一个 ELF 二进制数据（binary）的对齐需求的。它被编码在 ELF 的程序头（header）。在这个例子中，一个 x86-64 的二进制数据，值为 $200000_{16} = 2,097,152 = \text{2MB}$，与处理器所支持的最大页大小相符。

使用较大的页大小有个次要的影响：页表树的层次数量会被减低。由于对应到页偏移量的虚拟地址部分增加了，就没有剩下那么多需要通过页目录处理的 bit 了。这表示，在一次 TLB 错失的情况下，必须完成的工作总量减少了。

除了使用大页尺寸外，也可能通过将同时用到的数据搬移到较少的页上，以减少所需的 TLB 项目数量。这类似于我们先前讨论的针对 cache 使用的一些优化。
