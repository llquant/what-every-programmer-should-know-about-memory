# 4.3. 优化页表访问

页表的所有数据结构都会被保存在主内存中；操作系统就是在这里构建并更新表格的。在一个进程的创建、或是页表的一次修改之后，都会立即通知 CPU。页表是用于将每个虚拟地址，使用上述的页表走访来转成物理地址。更准确地说：每一层至少会有一个目录会被用在转换一个虚拟地址的过程中。这需要高达四次内存访问（以执行中进程的一个单一访问而言），这很慢。将这些目录表的项目视为普通的数据、并在 L1d、L2、等等 cache 它们是办得到的，但这可能还是太慢了。

从最早期的虚拟内存开始，CPU 设计者便已采用了一种不同的优化。一个简单的计算可以显示，仅将目录表的项目保存在 L1d 以及更高层次的 cache 中会招致可怕的性能。每个独立的地址计算会需要相符于页表深度的若干 L1d 访问。这些访问无法并行化，因为它们都依赖于前一次查询的结果。单是这样就会——在一台具有四个页表层级的机器上——需要至少 12 个周期。再加上 L1d 未命中的概率，结果是指令管道无法隐藏任何东西。额外的 L1d 访问也需要将宝贵的带宽偷到 cache 去。

所以，不只是将目录表的项目 cache 起来，而是连物理页地址的完整计算结果也会被 cache。跟代码与数据 cache 行得通的理由相同，这种 cache 的地址计算结果是很有效的。由于虚拟地址的页偏移量的部分不会参与到物理页地址的计算，仅有虚拟地址的剩余部分会用来作为 cache 的标签。视页大小而定，这代表数百或数千条的指令或数据对象会共享相同的标签，因而共享相同的物理地址前缀（prefix）。

存储计算得来的值的 cache 被称为转换后备缓冲区（Translation Look-Aside Buffer，TLB）。它通常是个很小的 cache，因为它必须非常快。现代的 CPU 提供了多层 TLB cache，就如同其他 cache 一样；更高层的 cache 更大也更慢。L1TLB 的小容量通常通过令 cache 为全相联、加上 LRU 逐出策略来弥补。近来，这种 cache 的大小已经持续成长，并且——在进行中——被转变为组相联。因此，当一个新的项目必须被加入时，被逐出并取代的项目可能不是最旧的一个。

如同上面所注记的，用来访问 TLB 的标签为虚拟地址的一部分。如果在 cache 中有比对到标签，最终的物理地址就可以通过将来自虚拟地址的页偏移量加到被 cache 的值上而计算出来。这是个非常快的过程；它必须如此，因为物理地址必须可用于每条使用独立地址的指令、以及——在某些情况下——使用物理地址作为键值（key）的 L2 查询。如果 TLB 查询未命中，处理器必须要进行一次页表走访；这可能是非常昂贵的。

通过软件或硬件预取代码或数据时，如果地址在另一个页上，可以暗自预取 TLB 的项目。这对于硬件预取而言是不可能的，因为硬件可能会引发无效的页表走访。程序员因而无法依赖硬件预取来预取 TLB 项目。必须明确地使用预取指令来实现。TLB——就像是数据与指令 cache——可以出现在多个层次。就如同数据 cache 一样，TLB 通常有两种：指令 TLB（ITLB）以及数据 TLB（DTLB）。像是 L2TLB 这种更高层次的 TLB 通常是统一式的，与其他 cache 的情况相同。
