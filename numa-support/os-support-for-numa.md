# 5.2.操作系统对 NUMA 的支持

为了支持 NUMA 机器，操作系统必须将内存分散式的性质纳入考量。举例来说，若是一个进程执行在一个给定的处理器上，被指派给进程地址空间的物理 RAM 理想上应该要来自本地内存。否则每条指令都必须为了代码与数据去访问远程的内存。有些仅存于 NUMA 机器的特殊情况要被考虑进去。DSO 的文字区段（text segment）在一台机器的物理 RAM 中通常正好出现一次。但若是 DSO 被所有 CPU 上的进程与线程用到（例如，像 `libc` 这类基本的运行期函数库），这表示并非一些、而是全部的处理器都必须拥有远程的地址。操作系统理想上会将这种 DSO「镜像（mirror）」到每个处理器的物理 RAM 中，并使用本地的副本。这并非一种优化，而是个必要条件，而且通常难以实现。它可能没有被支持、或者仅以有限的形式支持。

为了避免情况变得更糟，操作系统不该将一个进程或线程从一个节点迁移到另一个。操作系统应该已经试着避免在一般的多处理器机器上迁移进程，因为从一个处理器迁移到另一个处理器意味着 cache 内容遗失。若是负载分配需要将一个进程或线程迁出一个处理器，操作系统通常可以挑选任一个拥有足够剩余容量的新处理器。在 NUMA 环境中，新处理器的选择受到稍微多一些的限制。对于进程正在使用的内存，新选择的处理器不该有比旧的处理器还高的访问成本；这限制目标的清单。若是没有符合可用标准的空闲处理器，操作系统除了迁移到内存访问更为昂贵的处理器以外别无他选。

在这种情况下，有两种可能的方法。首先，可以期盼这种情况是暂时的，而且进程可以被迁回到一个更合适的处理器上。或者，操作系统也可以将进程的内存迁移到更靠近新使用的处理器的物理页上。这是个相当昂贵的操作。可能得要复制大量的内存，尽管不必在一个步骤中完成。当发生这种情况的时候，进程必须 –– 至少短暂地 –– 中止，以正确地迁移对旧页的修改。有了让页迁移高效又快速，有整整一串其他的必要条件。简而言之，操作系统应该避免它，除非它是真的有必要的。

一般来说，不可以假设在一台 NUMA 机器上的所有进程都使用等量的内存，使得 –– 由于遍及各个处理器的进程的分布 –– 内存的使用也会被均等地分配。事实上，除非执行在机器上的应用程序非常特殊（在 HPC 世界中很常见，但除此之外则否），不然内存的使用是非常不均等的。某些应用程序会使用巨量的内存，其余的几乎不用。若总是分配产生请求的处理器本地的内存，这将会 –– 或早或晚 –– 造成问题。系统最终将会耗尽执行大进程的节点本地的内存。

为了应对这些严重的问题，内存 –– 默认情况下 –– 不会只分配给本地的节点。为了利用所有系统的内存，默认策略是条带化（stripe）内存。这保证所有系统内存的同等使用。作为一个副作用，有可能变得能在处理器之间自由迁移进程，因为 –– 平均而言 –– 对于所有用到的内存的访问成本没有改变。由于很小的 NUMA 因子，条带化是可以接受的，但仍不是最好的（见 5.4 节的数据）。

这是个帮助系统避免严重问题、并在普通操作下更为可以预测的劣化（pessimization）。但它降低整体的系统性能，在某些情况下尤为显着。这就是 Linux 允许每个进程选择内存分配规则的原因。一个进程可以为它自己以及它的子进程选择不同的策略。我们将会在第六节介绍能用于此的接口。
